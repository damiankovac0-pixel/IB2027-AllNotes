# A1.4.1 Interpreters And Compilers

Translation changes source code into machine-executable instructions.

The main translation methods are:

- interpreters
- compilers
- just-in-time compilation
- bytecode interpreters

## Machine Code

Machine code, machine language, and machine instructions mean the same general idea: the binary instructions directly executed by a CPU.

Machine instructions depend on the processor architecture, such as x86 or ARM.

## Assembly Language

Assembly language is a low-level language that represents machine instructions with short readable mnemonics.

Examples:

- `MOV` for move
- `ADD` for add
- `LDR` for load
- `STR` for store

Assembly is more readable than raw binary, but it is still closely tied to hardware.

## Source Code

Source code is the human-readable program written in a high-level language such as Python, Java, C++, or JavaScript.

The computer cannot directly execute high-level source code without translation.

## Interpreter

An interpreter translates and executes code as the program runs, often line by line or instruction by instruction.

Examples of interpreted or commonly interpreted languages include Python and JavaScript, depending on environment.

Advantages:

- fast development feedback
- easy to test small changes
- useful for scripting and prototyping
- often more portable if the interpreter exists on the platform

Disadvantages:

- usually slower at execution than compiled code
- errors may only appear when that part of the code runs
- requires an interpreter to be installed

Best for:

- rapid development
- automation scripts
- scientific experimentation
- prototypes
- educational programming

## Compiler

A compiler translates the whole source code into machine code before execution.

The result is usually an executable file.

Examples of compiled languages include C, C++, Rust, and Go.

Advantages:

- faster execution after compilation
- many errors detected before the program runs
- executable can run without translating again
- suitable for performance-critical software

Disadvantages:

- compilation takes time
- less immediate feedback during development
- executable may be platform-specific
- different builds may be needed for different operating systems or hardware

Best for:

- operating systems
- games
- embedded systems
- performance-critical applications
- software where speed and efficiency matter

## Interpreted vs Compiled

| Factor | Interpreter | Compiler |
| --- | --- | --- |
| Translation time | during execution | before execution |
| Execution speed | usually slower | usually faster |
| Error detection | often at runtime | often before execution |
| Development speed | fast feedback | slower compile-run cycle |
| Portability | needs interpreter | executable may be platform-specific |
| Best use | scripting, prototyping | high-performance software |

## Just-In-Time Compilation

Just-in-time compilation, or JIT, is a hybrid approach.

It compiles code into machine code at runtime, just before execution or when a section is needed.

The idea is to combine:

- some development flexibility of interpretation
- some execution speed of compilation

Frequently used code sections can be compiled once and cached, improving performance.

Examples where JIT is common:

- JavaScript engines
- Java virtual machines
- .NET languages
- PyPy for Python

Advantages:

- can be faster than pure interpretation
- can optimize based on actual runtime behaviour
- keeps some portability through intermediate code

Disadvantages:

- runtime overhead from compiling during execution
- more complex system
- first execution of code may be slower before optimization

## Bytecode Interpreter

A bytecode interpreter executes bytecode, which is an intermediate representation between source code and machine code.

Basic process:

```text
source code -> bytecode -> bytecode interpreter -> execution
```

Bytecode is usually more compact and lower-level than source code, but not tied to one specific CPU like machine code.

Advantages:

- more portable than platform-specific machine code
- faster than interpreting raw source code directly
- can be used with virtual machines

Disadvantages:

- still usually slower than fully compiled native code
- requires a compatible interpreter or virtual machine
- runtime errors may still occur during execution

## Parser

A parser analyses source code structure during translation.

It checks whether the code follows the grammar of the language and builds a structured representation that later stages of translation can use.

Example: a parser checks whether brackets, operators, statements, and expressions are arranged legally.

## Error Detection

Different translation methods detect errors at different times.

- Compilers often detect many syntax and type errors before execution.
- Interpreters may only detect errors when the problematic line or section runs.
- JIT and bytecode systems may detect some errors before execution and some during runtime.

## Portability

Portability means how easily software can run on different systems.

- Interpreted code can be portable if the interpreter is available.
- Compiled executables are often tied to a specific platform.
- Bytecode can be portable if a compatible virtual machine exists.

## Choosing A Translation Method

| Scenario | Suitable method | Reason |
| --- | --- | --- |
| Quick prototype | interpreter | fast feedback and easy testing |
| Game engine | compiler | high performance needed |
| Cross-platform app | bytecode/JIT | portable intermediate representation |
| Script automation | interpreter | simple, flexible, quick to modify |
| Performance-critical embedded software | compiler | efficient native code |

## Exam Pattern

When evaluating translation processes, compare execution speed, development speed, error detection, portability, memory/resource use, and suitability for the scenario.
