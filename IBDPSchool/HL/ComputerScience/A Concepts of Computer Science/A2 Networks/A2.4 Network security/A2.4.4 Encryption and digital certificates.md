# A2.4.4 Encryption And Digital Certificates

Encryption converts readable data into coded data so unauthorized users cannot understand it.

Readable data is called plaintext.

Encrypted data is called ciphertext.

Decryption converts ciphertext back into plaintext using the correct key.

## Why Encryption Is Used

Encryption protects:

- confidentiality of data in transit
- confidentiality of stored data
- login credentials
- payment information
- private communications
- business and personal data

Encryption does not stop all attacks. If an attacker steals the key or compromises the endpoint, encrypted data may still be exposed.

## Symmetric Encryption

Symmetric encryption uses the same secret key for encryption and decryption.

Process:

1. Sender and receiver share the same secret key.
2. Sender encrypts plaintext using the key.
3. Ciphertext is sent across the network.
4. Receiver decrypts ciphertext using the same key.

Advantages:

- fast
- efficient for large amounts of data
- less computationally demanding than asymmetric encryption

Disadvantages:

- the key must be shared securely first
- if the key is stolen, communication is compromised
- key management becomes difficult with many users

## Asymmetric Encryption

Asymmetric encryption uses a pair of mathematically related keys:

- public key: can be shared openly
- private key: must be kept secret

If data is encrypted with a public key, only the matching private key can decrypt it.

Process for confidential communication:

1. Bob publishes his public key.
2. Alice encrypts a message using Bob's public key.
3. Alice sends the ciphertext to Bob.
4. Bob decrypts it using his private key.

Advantages:

- solves the problem of sharing a secret key in advance
- supports secure communication over public networks
- enables digital signatures

Disadvantages:

- slower than symmetric encryption
- requires more processing power
- depends on protecting the private key

## Symmetric vs Asymmetric Encryption

| Feature | Symmetric encryption | Asymmetric encryption |
| --- | --- | --- |
| Keys | one shared secret key | public/private key pair |
| Speed | faster | slower |
| Key distribution | difficult at scale | public key can be shared openly |
| Common use | bulk data encryption | key exchange, certificates, signatures |
| Main risk | shared key is stolen | private key is stolen |

## Hybrid Encryption

Many real systems use both methods.

Example: HTTPS commonly uses asymmetric cryptography to help establish trust and agree on a session key, then uses symmetric encryption for the main data transfer because it is faster.

## Digital Certificates

A digital certificate is an electronic document that binds a public key to an identity.

It helps prove that a public key belongs to a person, organization, website, or device.

A certificate may contain:

- owner's identity
- public key
- issuing certificate authority
- expiry date
- digital signature of the certificate authority

## Certificate Authorities

A certificate authority, or CA, is a trusted organization that verifies identity and signs digital certificates.

Browsers and operating systems contain lists of trusted CAs.

If a website presents a certificate signed by a trusted CA, the browser can verify that the certificate is valid and belongs to the website.

## HTTPS Certificate Process

Simplified process:

1. Website owner creates a public/private key pair.
2. Website owner sends a certificate signing request to a CA.
3. The CA verifies the website owner's identity.
4. The CA signs and issues a digital certificate.
5. The website installs the certificate on its server.
6. A browser visits the website and checks the certificate.
7. If valid, the browser establishes an encrypted HTTPS connection.

HTTPS provides:

- encryption
- server authentication
- data integrity during transmission

## Digital Signatures

A digital signature proves that data came from the holder of a private key and has not been changed.

Signing process:

1. The sender creates a hash of the data.
2. The sender encrypts the hash with their private key.
3. The encrypted hash is attached as the digital signature.

Verification process:

1. The receiver decrypts the signature using the sender's public key.
2. The receiver hashes the received data.
3. The receiver compares both hashes.
4. If they match, the data has integrity and the signature is valid.

Digital signatures support:

- integrity
- authentication
- non-repudiation

## Hash Functions

A hash function converts input data into a fixed-size digest.

Good cryptographic hash functions should be:

- deterministic: same input gives same output
- sensitive to changes: tiny input change gives very different output
- one-way: hard to reverse
- collision-resistant: hard to find two inputs with the same output

Hash functions are used for:

- checking data integrity
- digital signatures
- password storage
- file verification

## Key Management

Encryption is only as secure as the keys.

Good key management includes:

- protecting private keys
- rotating keys when needed
- revoking compromised certificates
- limiting who can access keys
- storing keys securely
- backing up keys safely

If a private key is stolen, an attacker may impersonate the owner or decrypt protected data.

## Digital Certificate Uses

Digital certificates are used in:

- HTTPS websites
- SSH authentication
- VPN authentication
- code signing
- digitally signed documents
