# A3.1.1 Benefits And Limitations Of Relational Databases

Relational databases are powerful for structured data, but they are not ideal for every problem.

## Benefits

### Reduced Data Duplication

Data is split into separate related tables, so the same data does not need to be repeated many times.

Example: owner details are stored once in an OWNER table instead of repeated for every pet treatment.

### Reduced Redundancy

Redundancy means unnecessary repeated data.

Reducing redundancy saves storage and makes updates easier.

### Data Consistency

Data consistency means the same fact is represented in the same way throughout the database.

Relational databases improve consistency because repeated values are reduced.

Example: an address is changed once in the OWNER table instead of in many treatment records.

### Data Integrity

Data integrity means data remains accurate, valid, and reliable.

Relational databases support integrity using:

- primary keys
- foreign keys
- constraints
- validation rules
- transaction controls

### Easier Data Retrieval

Relational databases can be queried using SQL.

SQL makes it possible to retrieve, filter, sort, join, and aggregate data from multiple related tables.

### Concurrency Control

Concurrency control allows multiple users to access and modify data without corrupting it.

This matters in systems such as banking, booking systems, online shops, and school databases.

### Reliable Transaction Processing

Relational database systems support reliable transactions.

A transaction is a set of database operations that should be treated as one complete unit.

Example: transferring money should subtract from one account and add to another. It should not do only one of those actions.

### Security

Relational database systems often provide built-in security features such as:

- user accounts
- access permissions
- roles
- views
- auditing

This supports multiuser systems where different users need different access levels.

### Scalability Compared With Flat Files

Relational databases are usually easier to scale than flat-file databases because data is organized into related tables and can be indexed, queried, secured, and maintained more effectively.

### Community Support

Relational database systems are widely used, so there is strong documentation, tooling, training, and community support.

## Limitations

### Design Complexity

Relational databases require careful design.

Designers must identify entities, attributes, keys, relationships, cardinality, modality, constraints, and normalization.

Poor design can cause inefficient queries, data anomalies, or hard-to-maintain systems.

### Rigid Schema

A relational database schema defines tables, fields, data types, and relationships.

This structure is useful for consistency, but it can be inflexible when data changes frequently or does not fit a predictable structure.

### Unstructured Data Handling

Relational databases are less suitable for unstructured data such as:

- free-form text collections
- varied documents
- raw social media content
- multimedia metadata with inconsistent fields

If records do not share common attributes, a relational schema can be difficult to design.

### Big Data Scalability

Relational databases can scale well for many business systems, but they may not be the best model for very large, distributed, high-volume, or rapidly changing datasets used in big data analytics.

Other database models may be better for some big data workloads.

### Hierarchical Data Handling

Hierarchical data has tree-like relationships.

Relational databases can store hierarchical data, but traversal can become inefficient or complex when the hierarchy is deep.

Examples:

- file systems
- organizational charts
- category trees
- comment threads

### Object-Relational Impedance Mismatch

Object-relational impedance mismatch is the difficulty of mapping object-oriented programming structures to relational tables.

Examples of mismatch:

- objects can contain references to other objects
- inheritance does not map cleanly to tables
- complex object structures may need many joins
- program data types may not match database types exactly

This can make application development more complex.

## Benefits vs Limitations Summary

| Benefits | Limitations |
| --- | --- |
| reduces duplication | complex to design well |
| improves consistency | rigid schema |
| improves integrity | poor fit for unstructured data |
| supports SQL retrieval | may not suit all big data workloads |
| supports concurrency | hierarchical data can be awkward |
| supports transactions | mismatch with OOP models |
| supports security | schema changes can be difficult |

## Exam Pattern

For evaluation questions, avoid claiming relational databases are always best. Explain that they are strong for structured, related data requiring consistency and transactions, but weaker for unstructured, rapidly changing, hierarchical, or very large distributed datasets.
