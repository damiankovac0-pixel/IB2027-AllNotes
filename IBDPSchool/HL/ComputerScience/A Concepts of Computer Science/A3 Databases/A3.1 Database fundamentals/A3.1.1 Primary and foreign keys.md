# A3.1.1 Primary And Foreign Keys

Keys are fields used to identify records and create relationships between tables.

## Primary Key

A primary key is a field that uniquely identifies each record in a table.

Example:

| StudentID | Name | House |
| --- | --- | --- |
| S001 | Ana | Blue |
| S002 | Omar | Red |

`StudentID` is a suitable primary key because each student has a different value.

## Primary Key Rules

A primary key should:

- be unique for every record
- not be empty
- not change unnecessarily
- identify exactly one record

Names are usually poor primary keys because more than one person can have the same name.

## Foreign Key

A foreign key is a field in one table that refers to the primary key in another table.

Foreign keys create relationships between tables.

Example:

OWNER table:

| OwnerID | OwnerName |
| --- | --- |
| O01 | Alison |
| O02 | Theo |

PATIENT table:

| PatientID | PatientName | OwnerID |
| --- | --- | --- |
| P01 | Oskar | O01 |
| P02 | Kai | O02 |

`OwnerID` is the primary key in OWNER and a foreign key in PATIENT.

This links each patient to an owner.

## Referential Integrity

Referential integrity means foreign key values must match valid primary key values in the related table.

Example: if PATIENT.OwnerID is `O01`, there must be an owner with OwnerID `O01` in the OWNER table.

Referential integrity prevents records from pointing to non-existent data.

## Composite Key

A composite key uses two or more attributes together to uniquely identify a record.

Example: in a TREATMENT table, `PatientID` alone may not be unique because one patient can have many treatments. `PatientID` plus `TreatmentDate` may be unique if each patient can only have one treatment per day.

## Concatenated Key

A concatenated key is made by combining values from multiple attributes into one key.

Example: part of a first name, part of a surname, and a date of birth might be joined to create an identifier.

Concatenated keys can be useful, but they may still create duplicates if the chosen data is not truly unique.

## Candidate Key

A candidate key is any field or combination of fields that could uniquely identify records.

One candidate key is chosen as the primary key.

Example: a student table might have both `StudentID` and `SchoolEmail` as candidate keys. The database designer chooses one as the primary key.

## Why Keys Matter

Keys allow databases to:

- uniquely identify records
- link related data across tables
- avoid storing the same data repeatedly
- enforce referential integrity
- support accurate queries

## Exam Pattern

To explain primary and foreign keys, say that a primary key uniquely identifies a record, while a foreign key stores a primary key from another table to create a relationship.
