# A3.2.1 Database Schema

A database schema is a plan or blueprint that shows how data is organized and how relationships between data are managed.

It describes the structure of the database before or during implementation.

## What A Schema Shows

A schema may show:

- entities or tables
- attributes or fields
- primary keys
- foreign keys
- relationships
- constraints
- data types
- indexes
- storage details

The exact detail depends on the schema level.

## Conceptual Schema

A conceptual schema is a high-level model of the database.

It focuses on what data exists and how the main entities relate.

It usually includes:

- main entities
- main relationships
- important business rules

It usually does not include:

- exact data types
- indexes
- storage methods
- DBMS-specific implementation details

Example for a sales system:

- entities: Customer, Order, Product
- relationship: Customer places Order
- relationship: Order contains Product

## Logical Schema

A logical schema gives a more detailed design of the database structure.

It includes:

- tables
- attributes
- primary keys
- foreign keys
- relationships
- constraints
- data types

It is normally independent of a specific database management system.

Example: the logical schema may state that CUSTOMER has `CustomerID`, `CustomerName`, and `EmailAddress`, and that `CustomerID` is the primary key.

## Physical Schema

A physical schema describes how the database will actually be implemented in a specific DBMS.

It may include:

- exact field data types
- indexes
- files
- storage structures
- access methods
- table partitions
- DBMS-specific constraints
- views

Example: `ProductID INT PRIMARY KEY AUTO_INCREMENT` and an index on `ProductName`.

## Conceptual vs Logical vs Physical

| Schema | Main focus | Typical audience |
| --- | --- | --- |
| Conceptual | high-level entities and relationships | users, analysts, designers |
| Logical | detailed tables, keys, fields, relationships | database designers |
| Physical | implementation details for a DBMS | programmers, database administrators |

## Why Schemas Matter

Schemas help developers:

- communicate database structure clearly
- identify design mistakes early
- plan relationships before building tables
- maintain consistency across implementation
- separate business design from technical storage details

## Exam Pattern

Be ready to describe each schema level and explain that database design moves from abstract conceptual design toward physical implementation.
