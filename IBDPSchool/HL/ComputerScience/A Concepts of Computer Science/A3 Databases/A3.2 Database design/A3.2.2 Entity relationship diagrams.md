# A3.2.2 Entity Relationship Diagrams

An entity relationship diagram, or ERD, is a visual model showing entities, attributes, and relationships in a database.

ERDs help designers plan a database before it is built.

## ERD Components

An ERD may show:

- entities
- attributes
- primary keys
- foreign keys
- relationships
- cardinality
- modality

## Entity

An entity is something the database stores data about.

Examples:

- STUDENT
- COURSE
- CUSTOMER
- ORDER
- PRODUCT

Entities are usually implemented as tables.

## Attribute

An attribute is a field that describes an entity.

Example attributes for STUDENT:

- StudentID
- FirstName
- LastName
- Grade
- Email

## Relationship

A relationship shows how entities are connected.

Relationships are often named using verbs.

Examples:

- CUSTOMER places ORDER
- STUDENT enrols in COURSE
- VET treats PATIENT

## Cardinality

Cardinality describes the maximum number of records that can be associated between entities.

Main types:

- one-to-one
- one-to-many
- many-to-many

## Modality

Modality describes the minimum participation in a relationship.

It shows whether the relationship is:

- optional
- mandatory

Example: an order must belong to a customer, so the order-to-customer relationship is mandatory from the order side.

## Constructing An ERD

Steps:

1. Identify the required ERD level: conceptual, logical, or physical.
2. Identify entities.
3. Identify attributes and data types if required.
4. Identify primary keys.
5. Identify relationships using verbs.
6. Identify cardinality.
7. Identify modality.
8. Add foreign keys where needed for implementation.

## Many-To-Many In ERDs

A many-to-many relationship should usually be resolved using a linking table.

Example:

```text
STUDENT many-to-many COURSE
```

becomes:

```text
STUDENT one-to-many ENROLMENT many-to-one COURSE
```

The linking table stores foreign keys from both original tables.

## Why ERDs Are Useful

ERDs help:

- communicate a database design
- identify missing entities
- identify incorrect relationships
- spot many-to-many relationships before implementation
- reduce design errors before coding starts
- support maintenance by showing the database structure clearly

## Exam Pattern

For construction questions, show entities clearly, include keys if expected, label relationships, and make cardinality/modality unambiguous.
