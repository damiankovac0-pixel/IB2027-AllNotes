# A3.2.3 Relational Database Data Types

A data type defines what kind of value an attribute can store.

Choosing suitable data types improves validation, storage efficiency, indexing, and query performance.

## Common Data Types

| Data type | Meaning | Example |
| --- | --- | --- |
| CHAR | fixed-length text | country code, fixed ID |
| VARCHAR(n) | variable-length text up to n characters | name, email |
| INTEGER | whole number | quantity, age, score |
| REAL | number with decimal part | price, measurement |
| DATE | date value | `2026-06-02` |
| TIME | time value | `14:30:00` |
| BOOLEAN | true or false | paid, active, archived |

## CHAR

CHAR stores fixed-length text.

It is useful when values are always the same length.

Examples:

- two-letter country code
- fixed-length user ID
- fixed-length airport code

## VARCHAR

VARCHAR stores variable-length text up to a maximum length.

It is useful when text values vary in length.

Examples:

- names
- email addresses
- product descriptions

## INTEGER

INTEGER stores whole numbers.

Use it for values that should not have decimals.

Examples:

- number of items
- year group
- stock quantity

## REAL

REAL stores decimal numbers.

Examples:

- price
- mass
- distance
- temperature

For currency, real databases often use fixed precision decimal types to avoid rounding issues, but the IB syllabus commonly uses REAL as the decimal type.

## DATE And TIME

DATE stores calendar dates.

TIME stores time of day.

Using date/time types is better than storing dates as plain text because the database can sort, compare, and validate them correctly.

## BOOLEAN

BOOLEAN stores true or false.

Examples:

- IsPaid
- IsArchived
- HasPermission

## Why Data Types Matter

Data types affect:

- what values can be stored
- what operations can be performed
- how much storage is needed
- whether indexes are efficient
- whether calculations are possible
- whether validation is easier

Example: if price is stored as text, calculations require conversion and may produce errors.

## Choosing Data Types

Choose the type based on the real-world meaning of the attribute.

Examples:

- phone numbers should usually be text, not integers, because they may include leading zeroes, spaces, or country codes
- IDs may be CHAR or VARCHAR if they are codes, even if they contain digits
- quantity should be INTEGER
- price should be REAL or a decimal numeric type

## Exam Pattern

When asked to choose a data type, justify it using the kind of data stored and the operations needed on it.
