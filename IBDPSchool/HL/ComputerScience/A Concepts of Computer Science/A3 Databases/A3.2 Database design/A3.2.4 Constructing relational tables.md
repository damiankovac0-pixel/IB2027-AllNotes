# A3.2.4 Constructing Relational Tables

Constructing relational tables means converting a database design into tables with suitable attributes, keys, data types, and relationships.

## Table Construction Checklist

For each table, identify:

- table name
- attributes
- primary key
- foreign keys
- data types
- required fields
- constraints
- relationships to other tables

## Naming Tables And Fields

Use clear names that describe the data.

Examples:

- STUDENT
- TEACHER
- COURSE
- ENROLMENT

Good field names:

- StudentID
- FirstName
- FamilyName
- CourseID

Avoid vague field names such as `Data1` or `Info`.

## Choosing Primary Keys

Every table should have a primary key.

A primary key uniquely identifies each record.

Examples:

- StudentID in STUDENT
- TeacherID in TEACHER
- CourseID in COURSE

If no natural field is unique, add a surrogate key such as `StudentID` or `OrderID`.

## Adding Foreign Keys

Foreign keys implement relationships between tables.

Example: if each student has one destination, the STUDENT table may include `DestinationID` as a foreign key pointing to DESTINATION.

## Constructing A One-To-Many Relationship

For a one-to-many relationship, place the primary key from the one side into the many side as a foreign key.

Example:

```text
TEACHER(TeacherID, FirstName, FamilyName)
CLASS(ClassID, ClassName, TeacherID*)
```

`TeacherID*` is a foreign key in CLASS.

## Constructing A Many-To-Many Relationship

For a many-to-many relationship, create a linking table.

Example:

```text
STUDENT(StudentID, FirstName, FamilyName)
COURSE(CourseID, CourseName)
ENROLMENT(StudentID*, CourseID*)
```

ENROLMENT links students and courses.

## Choosing Data Types

Assign each attribute a suitable data type.

Example:

```text
STUDENT(
  StudentID CHAR(6) PRIMARY KEY,
  FirstName VARCHAR(20),
  FamilyName VARCHAR(30),
  Grade INTEGER,
  Email VARCHAR(100)
)
```

## Constraints

Constraints are rules that protect data integrity.

Examples:

- primary key: value must be unique and not empty
- foreign key: value must match a valid primary key in another table
- not null: value must be present
- unique: no duplicate values allowed
- check: value must meet a condition

## Common Table Design Mistakes

- using names as primary keys
- storing multiple values in one field
- repeating groups of fields, such as Phone1, Phone2, Phone3
- failing to add a linking table for many-to-many relationships
- using text for values that need calculations
- duplicating the same data across many tables
- forgetting foreign keys

## Exam Pattern

For table construction questions, show tables clearly, mark primary keys and foreign keys, choose suitable data types, and avoid repeated groups.
