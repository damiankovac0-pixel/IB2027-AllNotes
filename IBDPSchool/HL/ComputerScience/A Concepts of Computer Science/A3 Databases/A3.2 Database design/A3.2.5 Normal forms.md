# A3.2.5 Normal Forms

Normal forms are stages of database normalization.

Normalization organizes data to reduce duplication, improve integrity, and reduce anomalies.

## Why Normalization Is Needed

Poorly designed databases can cause:

- insertion anomalies
- deletion anomalies
- update anomalies
- repeated data
- inconsistent data
- difficult maintenance

## Unnormalized Form

Unnormalized data may contain repeated groups, multiple values in one field, and mixed data about different entities in one table.

Example problem:

```text
StudentID, StudentName, Club1, Club2, Club3, Teacher1, Teacher2
```

This is difficult to query and update.

## First Normal Form

A table is in first normal form, or 1NF, when:

- each field contains atomic values
- there are no repeating groups
- each record can be uniquely identified

Atomic means the value cannot sensibly be split into smaller useful parts for this database.

Example issue: storing multiple phone numbers in one field violates 1NF.

## Second Normal Form

A table is in second normal form, or 2NF, when:
- it is already in 1NF
- every non-key attribute depends on the whole primary key

2NF matters especially when a table has a composite key.

Problem example:

```text
ENROLMENT(StudentID, CourseID, StudentName, CourseName)
```

If the key is `StudentID + CourseID`, then `StudentName` depends only on StudentID and `CourseName` depends only on CourseID. These should be moved to separate tables.

## Third Normal Form

A table is in third normal form, or 3NF, when:

- it is already in 2NF
- no non-key attribute depends on another non-key attribute

This removes transitive dependencies.

Problem example:

```text
CLUB(ClubID, ClubTitle, TeacherID, TeacherName)
```

If `TeacherName` depends on `TeacherID`, not directly on `ClubID`, then teacher details should be moved into a TEACHER table.

## Dependency Types

Functional dependency means one attribute determines another.

Example: `StudentID` determines `StudentName`.

Partial dependency means a non-key attribute depends on part of a composite key.

Transitive dependency means a non-key attribute depends on another non-key attribute.

## Normal Form Summary

| Normal form | Requirement | Main problem removed |
| --- | --- | --- |
| 1NF | atomic values, no repeating groups | repeated groups |
| 2NF | 1NF and no partial dependency | dependency on part of composite key |
| 3NF | 2NF and no transitive dependency | non-key depending on non-key |

## Exam Pattern

For normal form questions, identify which rule is broken, then split the table so each entity has its own table and relationships are maintained using keys.
