# A3.2.6 Normalizing To Third Normal Form

Normalizing to third normal form means transforming poorly organized data into tables that satisfy 1NF, 2NF, and 3NF.

## Overall Process

1. Start with the unnormalized data.
2. Identify repeated groups and make values atomic.
3. Identify entities.
4. Choose primary keys.
5. Move partial dependencies into separate tables.
6. Move transitive dependencies into separate tables.
7. Add foreign keys to preserve relationships.

## Step 1: Identify Repeating Groups

Look for fields that repeat the same kind of data.

Examples:

- Book1, Book2, Book3
- Phone1, Phone2, Phone3
- Club1, Club2, Club3

Repeated groups should be moved into separate records or separate tables.

## Step 2: Make Values Atomic

Each field should contain one value.

Bad:

```text
FullName = "Ana Kovac"
Subjects = "Math, CS, Physics"
```

Better if the database needs separate access:

```text
FirstName = "Ana"
FamilyName = "Kovac"
```

For repeating subjects, use a linking table rather than one comma-separated field.

## Step 3: Create 1NF

To reach 1NF:

- remove repeating groups
- ensure values are atomic
- identify a primary key

If needed, create a new table for the repeated data.

## Step 4: Create 2NF

To reach 2NF:

- start from 1NF
- check composite keys
- remove fields that depend on only part of a composite key

Example:

```text
ORDER_LINE(OrderID, ProductID, ProductName, Quantity)
```

If the key is `OrderID + ProductID`, then `ProductName` depends only on ProductID. Move product details to PRODUCT.

## Step 5: Create 3NF

To reach 3NF:

- start from 2NF
- remove fields that depend on other non-key fields

Example:

```text
STUDENT(StudentID, StudentName, TutorID, TutorName)
```

`TutorName` depends on `TutorID`, not directly on `StudentID`, so move tutor details into TUTOR.

## Step 6: Preserve Relationships

When data is split into new tables, add foreign keys so the original meaning is not lost.

Example:

```text
STUDENT(StudentID, StudentName, TutorID*)
TUTOR(TutorID, TutorName)
```

The foreign key `TutorID*` links STUDENT to TUTOR.

## Normalization Checklist

- Does each table represent one entity?
- Does each table have a primary key?
- Are all values atomic?
- Are repeated groups removed?
- Does every non-key field depend on the whole key?
- Does every non-key field depend only on the key?
- Are foreign keys included to maintain relationships?

## Common Exam Approach

When given a flat-file table:

1. Circle repeated data.
2. Identify likely entities.
3. Assign primary keys.
4. Move related fields into their entity tables.
5. Add foreign keys.
6. Check that no table contains unrelated fields.

## Exam Pattern

In construction questions, marks often come from correct tables, correct primary keys, correct foreign keys, and removing the correct dependencies. Show your design clearly.
