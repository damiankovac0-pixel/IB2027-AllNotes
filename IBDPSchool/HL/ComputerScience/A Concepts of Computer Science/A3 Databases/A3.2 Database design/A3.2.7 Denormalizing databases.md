# A3.2.7 Denormalizing Databases

Denormalization deliberately adds redundancy to a previously normalized database to improve performance or simplify queries.

It is not the same as poor database design. It is a deliberate tradeoff.

## Why Denormalize

Highly normalized databases often require joins to combine data from many tables.

If a system is queried heavily, repeated joins can slow performance.

Denormalization can store commonly accessed data together so queries need fewer joins.

## Example

In an online shop, product details and artist details may be stored in separate normalized tables.

If product pages almost always show artist name with product details, the artist name might be copied into the product display table to avoid repeated joins.

## Advantages Of Denormalization

- faster read queries
- fewer joins
- simpler queries
- improved reporting performance
- useful for read-heavy systems
- can reduce load on database servers

## Disadvantages Of Denormalization

- duplicated data
- increased storage use
- greater risk of update anomalies
- more complex updates
- possible data inconsistency
- harder maintenance

## When Denormalization May Be Useful

Denormalization may be justified when:

- the database is read-heavy
- query speed is more important than storage efficiency
- reports repeatedly need the same joined data
- joins are causing unacceptable performance problems
- duplicated fields are carefully controlled

## When Denormalization Is Risky

Denormalization is risky when:

- data changes frequently
- consistency is critical
- update logic is complex
- duplicated values may become inconsistent
- the design is being used to hide poor normalization skills

## Balancing Normalization And Denormalization

Good database design balances:

- data integrity
- query performance
- storage requirements
- maintenance complexity
- update frequency
- application requirements

A partially denormalized schema may be the best compromise for some systems.

## Mitigating Denormalization Risks

If denormalization is used, risk can be reduced by:

- clear documentation
- validation rules
- controlled update procedures
- triggers or application logic to keep copies synchronized
- monitoring query performance
- reviewing storage growth

## Normalization vs Denormalization

| Normalization | Denormalization |
| --- | --- |
| reduces duplication | deliberately adds duplication |
| improves consistency | may risk inconsistency |
| reduces anomalies | may introduce update anomalies |
| often needs more joins | often needs fewer joins |
| better for frequent updates | better for heavy reads and reporting |

## Exam Pattern

For evaluation questions, explain the tradeoff: denormalization can improve read performance and simplify queries, but increases redundancy, storage use, and the risk of inconsistency.
