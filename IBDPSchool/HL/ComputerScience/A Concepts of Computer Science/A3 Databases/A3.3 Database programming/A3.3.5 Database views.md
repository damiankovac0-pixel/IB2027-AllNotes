# A3.3.5 Database Views

This is an HL syllabus point.

A database view is a virtual table based on the result of a SELECT query.

Views do not normally store the data themselves. They present data from one or more tables in a customized way.

## Creating A View

Example:

```sql
CREATE VIEW EmployeeDetails AS
SELECT Employee.EmployeeID,
       Employee.LastName,
       Department.DepartmentName
FROM Employee
JOIN Department ON Employee.DepartmentID = Department.DepartmentID;
```

The view can then be queried like a table:

```sql
SELECT *
FROM EmployeeDetails;
```

## Why Views Are Used

Views can:

- simplify complex queries
- hide unnecessary columns
- present data in a user-friendly way
- restrict access to sensitive fields
- combine data from multiple tables
- support reporting
- provide a consistent interface for users or applications

## Security Benefit

Views can limit what users see.

Example: a payroll table may include salary, bank details, and tax identifiers. A view for managers might show employee names and departments but hide bank details.

## Simpler User Access

Database users may not know how to write joins.

A view lets developers define the join once, then users can query the view more simply.

## Types Of Views

Common view purposes:

- simple view: based on one table
- complex view: based on multiple tables, joins, or calculations
- security view: hides sensitive fields or rows
- reporting view: organizes data for reports or dashboards

## Limitations Of Views

Views can have limitations:

- complex views may still be slow because the underlying query must run
- not all views are updatable
- changes to underlying tables can break views
- too many views can make a database harder to understand

## View vs Table

| Feature | Table | View |
| --- | --- | --- |
| Stores data | yes | usually no |
| Based on query | no | yes |
| Used for | actual data storage | presentation, security, simplification |
| Can combine tables | only by query | yes, through the view definition |
