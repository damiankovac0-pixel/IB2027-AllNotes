# A3.3.6 Database Transactions And ACID

This is an HL syllabus point.

A transaction is a sequence of database operations treated as one complete unit of work.

Transactions maintain data integrity when databases are accessed by multiple users or when operations involve multiple related changes.

## Why Transactions Are Needed

Many database operations must be completed fully or not at all.

Example: transferring money between accounts requires two updates:

1. subtract money from account A
2. add money to account B

If only one update happens, the database becomes incorrect.

## ACID Properties

ACID describes the four key properties of reliable transactions:

- atomicity
- consistency
- isolation
- durability

## Atomicity

Atomicity means a transaction is indivisible.

Either all operations in the transaction complete successfully, or none of them do.

If part of the transaction fails, the database rolls back to its previous state.

## Consistency

Consistency means a transaction moves the database from one valid state to another valid state.

All rules, constraints, and relationships must still be satisfied after the transaction.

Example: a foreign key cannot be left pointing to a non-existent primary key.

## Isolation

Isolation means transactions do not interfere with each other while running concurrently.

This prevents users from seeing or creating inconsistent intermediate data.

Example: two employees should not both sell the final available train ticket at the same time.

## Durability

Durability means that once a transaction has been committed, its changes are permanent.

Committed data should survive system failure, power loss, or restart.

## COMMIT

COMMIT permanently saves a successful transaction.

Example:

```sql
BEGIN TRANSACTION;
UPDATE Ticket SET Status = 'sold' WHERE TicketID = 1001;
INSERT INTO Sale (TicketID, CustomerID) VALUES (1001, 55);
COMMIT;
```

## ROLLBACK

ROLLBACK cancels a transaction and restores the database to its previous valid state.

Example:

```sql
BEGIN TRANSACTION;
UPDATE Account SET Balance = Balance - 100 WHERE AccountID = 1;
UPDATE Account SET Balance = Balance + 100 WHERE AccountID = 2;
ROLLBACK;
```

## Concurrent Access Problem

Databases are useful because multiple users can access them at the same time.

However, concurrent access can cause problems such as:

- lost updates
- dirty reads
- inconsistent reads
- double booking
- incorrect stock counts

Transactions and isolation controls reduce these risks.

## ACID Summary

| Property | Meaning | Protects against |
| --- | --- | --- |
| Atomicity | all or nothing | partial updates |
| Consistency | valid state to valid state | broken rules and constraints |
| Isolation | concurrent transactions do not interfere | race conditions and inconsistent reads |
| Durability | committed changes persist | data loss after commit |

## Exam Pattern

For transaction questions, define transaction and ACID. Use a scenario such as booking, banking, ticket sales, or stock control to explain why all-or-nothing and isolation are necessary.
